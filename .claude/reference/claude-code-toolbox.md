# Claude Code 기능 — 언제 무엇을 쓰나

"이런 동작이 필요한데 어떤 기능을 쓰면 좋지?" 에 답하기 위한 참조표.
각 행은 `기능 | 언제 쓰면 좋은가 | 안 써도 될 때` 3열.

> 출처: Anthropic Claude Code best-practices, martinfowler.com/articles/harness-engineering.html (Agent = Model + Harness),
> deep-research(2026-06). 하네스 = "guides(사전 유도: 지침·계획·Skill) + sensors(사후 피드백: 테스트·린터·리뷰)".

## 1. 탐색 · 이해 (코드/구조 파악)

| 기능 | 언제 쓰면 좋은가 | 안 써도 될 때 |
|---|---|---|
| **Explore 에이전트** | 여러 파일·디렉토리를 훑어 결론만 필요할 때. 컨텍스트를 메인 세션에 안 쌓고 탐색. | 어느 파일·심볼인지 이미 알 때 (직접 읽기). |
| **Grep / Glob** | 키워드·패턴·파일명으로 위치를 빨리 찾을 때. | 전체 흐름 이해가 필요할 때(에이전트로 위임). |
| **`@파일` 참조** | 특정 파일을 컨텍스트에 명시적으로 넣을 때. | 파일이 많아 일일이 못 넣을 때(에이전트 탐색). |
| **code-explorer 에이전트** | 기존 기능의 실행 경로·아키텍처를 깊게 추적해야 할 때. | 단순 위치 확인(Grep으로 충분). |

## 2. 계획 · 설계 (구현 전)

| 기능 | 언제 쓰면 좋은가 | 안 써도 될 때 |
|---|---|---|
| **Plan mode / Plan 에이전트** | 구현 전 전략을 세우고 검토받고 싶을 때. "엉뚱한 문제를 푸는 것"을 방지. | diff를 한 문장으로 설명 가능한 사소한 작업(생략 권장). |
| **brainstorming 스킬** | 여러 결정이 얽히거나 되돌리기 어려운 작업의 의도·요구사항 정리. | 요구사항이 이미 명확하고 단순할 때. |
| **writing-plans 스킬** | 멀티스텝 작업을 단계별 구현 계획으로 옮길 때. | 단일 단계로 끝나는 작업. |
| **AskUserQuestion** | 사용자만 정할 수 있는 결정에서 막혔을 때(기본값으로 못 메움). | 코드/관례로 합리적 기본값을 정할 수 있을 때. |
| **harness 스킬 (이 하네스)** | 1작업=1문서=1커밋으로 기록을 남기며 진행할 때. | 오타·상수 1개 수정 등 trivial. |

## 3. 구현 · TDD

| 기능 | 언제 쓰면 좋은가 | 안 써도 될 때 |
|---|---|---|
| **test-driven-development (실패 테스트 먼저)** | 검증 가능한 기능/버그픽스. **단, 어떤 테스트/파일이 위험한지 맥락을 함께 줄 것** — 절차만 주면 오히려 악화. | 검증 기준을 세우기 어려운 탐색적 작업. |
| **RED 강제 ("구현하지 마라")** | 모델이 기본적으로 구현부터 쓰므로, 진짜 test-first가 필요할 때 명시적으로 박는다. | 테스트가 무의미한 설정/문서 작업. |
| **작은 diff** | 거의 항상. 리뷰·롤백·디버깅이 쉬워짐. | (예외 거의 없음) |
| **subagent-driven-development** | 독립적인 작업들을 같은 세션에서 서브에이전트로 나눠 진행할 때. | 순차 의존이 강한 단일 흐름. |

## 4. 검증 · 리뷰 (sensors)

| 기능 | 언제 쓰면 좋은가 | 안 써도 될 때 |
|---|---|---|
| **검증 커맨드 제공 (테스트/빌드/린터)** | 거의 항상. 없으면 "looks done"이 유일 신호가 되어 사람이 검증 루프가 됨. | 검증을 만들 수 없는 순수 탐색. |
| **verification-before-completion 스킬** | "완료/통과" 라고 주장하기 전. 실제 출력으로 증거를 확보. | 아직 작업 중일 때. |
| **/code-review** | diff의 버그·중복·단순화 여지를 점검할 때. effort로 깊이 조절. | 변경이 사소하고 자명할 때. |
| **requesting / receiving-code-review 스킬** | 큰 기능 완료·머지 전, 또는 리뷰 피드백을 받을 때. | 혼자 끝내는 trivial 변경. |
| **Stop hook (verify-before-commit)** | 양보 불가 invariant("커밋 전 검증 통과")를 **보장**해야 할 때. 마크다운 권고는 무시될 수 있음. | invariant가 없거나 검증이 불가능한 프로젝트. |

## 5. 병렬 · 규모

| 기능 | 언제 쓰면 좋은가 | 안 써도 될 때 |
|---|---|---|
| **dispatching-parallel-agents 스킬** | 공유 상태·순차 의존 없는 독립 작업 2개 이상을 동시에. | 작업들이 서로 의존하거나 1개뿐일 때. |
| **Workflow (멀티에이전트 오케스트레이션)** | 광범위 감사·마이그레이션·다관점 리뷰 등 한 컨텍스트로 못 담는 규모. **사용자가 명시적으로 opt-in 했을 때만.** | 일반 작업. 토큰 비용이 큼. |
| **백그라운드 실행 (run_in_background)** | 오래 걸리는 빌드/테스트/CI를 기다리지 않고 진행할 때. | 즉시 끝나는 커맨드. |

## 6. 기록 · 자동화

| 기능 | 언제 쓰면 좋은가 | 안 써도 될 때 |
|---|---|---|
| **Skill / 슬래시 커맨드** | 가끔만 필요한 상황별 워크플로우·도메인 지식. 온디맨드 로드. | 항상 적용되는 넓은 규칙(→ CLAUDE.md). |
| **CLAUDE.md / AGENTS.md** | 매 세션 적용되는 **넓은** 규칙만. 비대해지면 모델이 실제 지침을 무시함. | 가끔만 쓰는 절차(→ Skill로). |
| **hooks** | 자동 동작("X 할 때마다 Y")을 **결정적으로 보장**할 때. 모델이 아니라 하네스가 실행. | 권고로 충분한 유연한 지침. |
| **memory** | 세션을 넘겨 기억해야 할 사용자 선호·프로젝트 맥락. | 이번 대화에만 의미 있는 것. |
| **/loop · /schedule** | 반복 작업·인터벌 폴링·예약 실행. | 일회성 작업. |

## 7. 컨텍스트 관리

| 기능 | 언제 쓰면 좋은가 | 안 써도 될 때 |
|---|---|---|
| **서브에이전트로 위임** | 큰 탐색·읽기를 메인 컨텍스트에 안 쌓고 결론만 받을 때. (컨텍스트는 가장 중요한 자원) | 결과를 직접 단계별로 봐야 할 때. |
| **worktree 격리** | 여러 에이전트가 파일을 병렬 수정해 충돌 위험이 있을 때. | 단일 작업(셋업 비용만 듦). |
| **trivial은 그냥 처리** | 사소한 작업에 무거운 계획/리뷰를 붙이지 않기. | (오버헤드가 정당한 큰 작업) |

## 빠른 결정 가이드

- **"엉뚱한 걸 만들까 걱정"** → 계획 먼저 (Plan / brainstorming).
- **"제대로 됐는지 모르겠다"** → 검증 커맨드 + verification-before-completion.
- **"매번 까먹지 않게 강제하고 싶다"** → hook (권고가 아니라 보장).
- **"가끔만 쓰는 절차다"** → Skill (CLAUDE.md에 넣지 말 것).
- **"독립 작업이 여러 개다"** → 병렬 에이전트.
- **"한 컨텍스트로 안 담긴다"** → Workflow (단, opt-in).
